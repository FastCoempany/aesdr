TASK 1.01

Session Expiry During Lesson

Supabase token expires mid-lesson, progress saves fail silently.

Step-by-Step Execution

  1. Test what happens when session expires mid-lesson
    • User is in iframe, session token expires (default 1 hour)
    • postMessage bridge fires, server action fails silently
    • User clicks "Mark Complete" and gets cryptic error
  2. Mitigation
    • Add session refresh logic in ProgressSaver component
    • Show "Session expired — please sign in again" toast instead of silent fail
    • Preserve lesson position in localStorage as backup
  3. Test the fix
    • Manually expire session in Supabase dashboard
    • Verify graceful error message appears

Done When

  • Session expiry shows clear user-facing message
  • Progress is not silently lost

Common Mistakes

  • Refreshing token too aggressively (unnecessary API calls)
  • Redirecting to login mid-lesson (loses context)
TASK 1.02

Shared Team Account Abuse

Team tier could be shared via one login with no seat tracking.

Step-by-Step Execution

  1. Identify the risk
    • Team tier ($999 for 10 seats) shared via one login
    • No seat tracking = unlimited users on one account
  2. Mitigation options
    • Track unique IPs/devices per user_id in course_progress
    • Add team_members table linking team_id to individual accounts
    • Flag accounts with 10+ distinct IPs in 24 hours
  3. Decision
    • Not worth building before launch — monitor first
    • Revisit after first team sale

Done When

  • Risk documented and acknowledged
  • Monitoring plan in place for post-launch

Common Mistakes

  • Over-engineering DRM before having any team customers
TASK 1.03

URL Manipulation for Premium Content

Users could guess /course/5 URL and access lessons without paying.

Step-by-Step Execution

  1. Identify the risk
    • Currently proxy.ts blocks all non-public paths
    • After launch, course pages only check auth — not purchase status
    • Any signed-up user could access lessons without paying
  2. Audit the flow
    • app/course/[lessonId]/page.tsx checks Supabase auth only
    • No purchase verification exists yet
  3. Mitigation (after Stripe integration)
    • Add purchase check: query purchases table for user_id

Done When

Common Mistakes

TASK 1.04

PDF Tool Download Bypass

Tool HTML files served at /tools/[slug] have no auth check — only the /download route is gated.

Step-by-Step Execution

TASK 2.01

Supabase Row-Level Security Gaps

Without RLS, any authenticated user can read ALL users' progress data.

Step-by-Step Execution

  1. Verify RLS is enabled on course_progress table
    • Check Supabase dashboard: Tables > course_progress > RLS toggle
    • Verify policy: users can only SELECT/INSERT/UPDATE their own rows
    • Without RLS, any auth user can read all progress
  2. Verify RLS on future tables
    • purchases table (when created) needs RLS
    • team_members table (when created) needs RLS
  3. Test
    • Log in as User A, try to query User B's progress
    • Should return empty results, not data

Done When

  • RLS enabled and policies verified on all tables
  • Cross-user data access returns empty

Common Mistakes

  • Enabling RLS without adding policies (blocks ALL access)
  • Forgetting to add RLS to new tables

TASK 2.03

Backup Strategy

Know what's backed up, where, and how to recover.

Step-by-Step Execution

  1. Supabase automatic backups
    • Free tier: daily backups, 7-day retention
    • Pro tier: point-in-time recovery
    • Know which tier you're on
  2. Content backups
    • 36 lesson HTML files in git — covered
    • Tool HTML files in git — covered
    • User data in Supabase — covered by their backups
  3. Disaster recovery
    • Supabase outage: site shows empty progress, doesn't crash
    • Vercel outage: nothing you can do — accept the dependency
    • Git repo: push to GitHub — that's your backup

Done When

  • Backup status documented for each data store
  • Recovery plan exists for each failure scenario

Common Mistakes

  • Assuming backups exist without verifying
  • Not testing restore from backup at least once

TASK 3.01

Lesson HTML Injection via postMessage

If lesson HTML is modified, malicious postMessages could be sent to the parent.

Step-by-Step Execution

  1. Identify the risk
    • Lesson HTML in iframes uses postMessage to communicate
    • If HTML is tampered, it could send malicious messages
  2. Current state
    • Check: does ProgressSaver.tsx verify event.origin?
    • Does it validate message structure?
  3. Hardening
    • Add origin whitelist to message listener
    • Validate message type is "aesdr:progress"
    • Sanitize stateData before storing in DB

Done When

  • postMessage listener validates origin and message structure
  • Invalid messages are silently rejected

Common Mistakes

  • Blocking legitimate messages by being too strict on origin
  • Logging rejected messages to console (information leak)
TASK 3.02

Browser Back Button Breaks Lesson State

Hitting back during a lesson may exit to dashboard instead of previous screen.

Step-by-Step Execution

    • iframe navigates: does it go to screen 6 or exit entirely?
  2. Test it
    • Open a lesson, navigate to screen 5, hit browser back
    • Document actual behavior
  3. Mitigation if needed
    • Lesson HTML manages its own state (go/back functions)
    • Browser back exits iframe entirely — expected behavior
    • Progress saved via postMessage on each screen change — no data loss

Done When

  • Behavior documented
  • No silent data loss on back button

Common Mistakes

  • Trying to intercept browser back (bad UX pattern)
